## 3. 测试用例编写指南

### 3.1 用例编号规则

**编号格式:** `[模块缩写]-[序号]`

**模块缩写对照表:**
| 模块 | 缩写 | 示例 |
|------|------|------|
| Albums API | ALB | ALB-001 |
| Timeline API | TL | TL-001 |
| AI Features | AI | AI-001 |
| Growth Tools | GR | GR-001 |
| Photo Upload | PU | PU-001 |
| Authentication | AUTH | AUTH-001 |
| Security | SEC | SEC-001 |

**示例:**
- ALB-001: Albums API - 创建相册
- TL-015: Timeline API - 创建里程碑
- AI-008: AI功能 - 照片质量评分

### 3.2 优先级定义

| 优先级 | 定义 | 示例 |
|--------|------|------|
| **P0** | 核心功能，阻塞上线 | 登录、上传照片 |
| **P1** | 重要功能，影响用户体验 | 创建相册、时间线查看 |
| **P2** | 次要功能，可短期延后 | 评论、分享美化 |
| **P3** | 边缘情况，可后续优化 | 错误提示文案、UI细节 |

### 3.3 测试类型定义

| 类型 | 说明 | 示例 |
|------|------|------|
| **功能测试** | 验证功能正确性 | 创建相册成功 |
| **性能测试** | 验证性能指标 | 响应时间 < 200ms |
| **安全测试** | 验证安全性 | SQL注入防护 |
| **集成测试** | 验证模块间交互 | 相册删除关联照片 |
| **UI/UX测试** | 验证界面和体验 | 按钮可点击、布局合理 |

### 3.4 测试步骤编写规范

**DO ✅:**
- 使用动宾结构: "点击" + "按钮"
- 步骤编号: 1, 2, 3...
- 每个步骤只做一件事
- 包含验证步骤
- 预期结果明确具体

**DON'T ❌:**
- "然后"、"接着"等模糊词
- 一个步骤多个操作
- 预期结果含糊
- 缺少验证步骤

**示例:**

❌ **不好:**
```
1. 点击创建按钮
2. 然后输入信息
3. 应该成功
```

✅ **好:**
```
1. 点击"创建相册"按钮
2. 输入相册名称"测试相册"
3. 点击"确定"按钮
4. 预期: 跳转到相册详情页，显示成功提示
```

### 3.5 测试数据规范

**测试数据应包含:**
- **正常数据**: 符合规范的有效数据
- **边界数据**: 最小值、最大值、临界值
- **异常数据**: 空值、非法值、超长值

**示例:**
```yaml
正常数据:
  - 相册名称: "测试相册" (20字符)
  - 描述: "这是一个测试相册" (100字符)

边界数据:
  - 相册名称: "测" (1字符，最小)
  - 相册名称: "测试相册123456789012345" (30字符，最大)

异常数据:
  - 相册名称: "" (空值)
  - 相册名称: "测试@#$%" (特殊字符)
  - 相册名称: "a" * 200 (超长，200字符)
```

## 4. 测试用例管理

### 4.1 用例生命周期

```
[创建] → [待执行] → [执行中] → [已完成]
                     ↓           ↓
                  [阻塞]      [失败] → [已修复] → [回归测试]
```

### 4.2 用例状态定义

| 状态 | 说明 | 标记 |
|------|------|------|
| **待执行** | 用例已创建，等待执行 | 📋 |
| **进行中** | 用例正在执行 | ⏳ |
| **已通过** | 用例执行成功 | ✅ |
| **已失败** | 用例执行失败 | ❌ |
| **阻塞** | 用例无法执行（环境/依赖） | 🚫 |

### 4.3 用例维护规则

**何时更新用例:**
- ✅ 新功能上线后，补充测试用例
- ✅ Bug修复后，增加回归测试用例
- ✅ 用户反馈问题后，增加场景用例
- ✅ 定期审查（每月）淘汰过时用例

**用例评审标准:**
- [ ] 步骤清晰可执行
- [ ] 预期结果明确
- [ ] 测试数据充分
- [ ] 覆盖正常+边界+异常
- [ ] 优先级合理
- [ ] 编号规范统一

## 7. 最佳实践

### 7.1 用例设计原则

1. **独立性**: 每个用例独立执行，不依赖其他用例
2. **可重复性**: 多次执行结果一致
3. **可维护性**: 用例结构清晰，易于修改
4. **充分性**: 覆盖正常、边界、异常情况
5. **可执行性**: 步骤明确，任何人都能执行

### 7.2 用例评审清单

**评审前准备:**
- [ ] 准备测试环境
- [ ] 准备测试数据
- [ ] 邀请相关人员（开发、产品）

**评审重点:**
- [ ] 用例覆盖是否充分
- [ ] 步骤是否清晰可执行
- [ ] 预期结果是否明确
- [ ] 优先级是否合理
- [ ] 是否有遗漏场景

**评审后行动:**
- [ ] 记录评审意见
- [ ] 更新用例
- [ ] 重新评审（如需要）

### 7.3 常见问题

**Q1: 用例颗粒度如何把握?**

**A:**
- ✅ 一个用例验证一个功能点
- ✅ 步骤不超过10步
- ❌ 一个用例覆盖多个功能

**Q2: 测试数据如何准备?**

**A:**
- ✅ 使用脚本自动生成
- ✅ 边界数据手工准备
- ✅ 生产数据脱敏后使用

**Q3: 用例数量多少合适?**

**A:**
- 功能测试: 每个功能5-10个用例
- 集成测试: 每个场景3-5个用例
- 安全测试: 每个漏洞类型2-3个用例
